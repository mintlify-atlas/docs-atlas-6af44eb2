## The Evolution Philosophy

Most starter templates lock you into decisions made at initialization. Aegis Stack is different: **your choices aren't permanent**.

Start with what you need today. Add components when requirements change. Remove what you outgrow. Stay current with template improvements.

## Lifecycle Commands

Aegis Stack provides three commands for managing your project over time:

### `aegis add` - Add Components

Add infrastructure components to an existing project.

```bash
# Add single component
aegis add scheduler

# Add multiple components
aegis add worker,database

# Add with configuration (scheduler backend)
aegis add scheduler[sqlite]

# Interactive selection
aegis add --interactive

# Add to specific project
aegis add worker --project-path ../my-api
```

From `aegis/commands/add.py:27-350`

<Info>
**Safe Operation**: Adding components never overwrites existing files. New files are added and configuration is updated in `.copier-answers.yml`.
</Info>

### `aegis remove` - Remove Components

Remove infrastructure components from an existing project.

```bash
# Remove single component
aegis remove scheduler

# Remove multiple components
aegis remove worker,redis

aegis remove scheduler --project-path ../my-api
```

### `aegis update` - Update Template Version

Update your project to a newer template version while preserving customizations.

```bash
# Update to latest (CLI version)
aegis update

# Update to specific version
aegis update --to-version 0.6.0

# Preview changes without applying
aegis update --dry-run

# Update specific project
aegis update --project-path ../my-api

# Use custom template
aegis update --template-path ~/workspace/aegis-stack
```

## Evolution Workflow

A typical evolution workflow across project lifetime:

### Week 1: Ship MVP

```bash
# Start minimal
aegis init my-api
cd my-api
uv sync
cp .env.example .env
make serve
```

Components: `backend`, `frontend` (core only)

### Week 3: Add Scheduled Reports

```bash
# Business needs periodic reports
aegis add scheduler
make serve
```

Components: `backend`, `frontend`, `scheduler`

### Month 2: Need Async Workers

```bash
# Long-running tasks blocking API
aegis add worker
make serve
```

Components: `backend`, `frontend`, `scheduler`, `worker`, `redis` (auto-added)

### Month 6: Scheduler Not Needed

```bash
# Moved scheduled tasks to worker
aegis remove scheduler
make serve
```

Components: `backend`, `frontend`, `worker`, `redis`

### Month 12: Stay Current

```bash
# Update to latest template improvements
aegis update
make check
git commit -m "Update Aegis Stack template to v0.7.0"
```

## Version Compatibility

### CLI and Template Versions

Aegis Stack maintains version compatibility between CLI and templates:

```bash
# Check versions
aegis --version  # CLI version
cat .copier-answers.yml | grep aegis_version  # Template version
```

From `aegis/core/version_compatibility.py`:

```python
def validate_version_compatibility(
    project_path: Path,
    command_name: str,
    force: bool = False
) -> None:
    """Validate CLI version matches project template version."""
    cli_version = get_cli_version()
    project_version = get_project_template_version(project_path)
    
    if cli_version != project_version and not force:
        # Show warning, suggest update
        pass
```

## Git-Aware Merging

Aegis Stack uses Copier's git integration for safe updates:

### How Git-Aware Updates Work

<Accordion title="Template as Git Repository">
The Aegis Stack template is a git repository. When you initialize a project, Copier records:

```yaml
# .copier-answers.yml
_src_path: gh:lbedner/aegis-stack
_commit: abc123def456  # Git commit used for generation
```

This enables Copier to:
1. Know exactly which template version generated your project
2. Fetch changes between template versions
3. Perform 3-way merges (base, template changes, your changes)
</Accordion>

<Accordion title="3-Way Merge Process">
When you run `aegis update`, Copier:

1. **Identifies base**: Original template at `_commit`
2. **Fetches changes**: New template at target version
3. **Detects customizations**: Your modifications to generated code
4. **Merges intelligently**:
   - Template changes in untouched files → Applied cleanly
   - Your changes in unchanged template areas → Preserved
   - Both changed same area → Conflict (`.rej` file created)

From `aegis/core/template_cleanup.py:sync_template_changes()`
</Accordion>

<Accordion title="Conflict Resolution">
When conflicts occur:

```bash
# Update creates .rej files
aegis update
   2 file(s) have merge conflicts (search for <<<<<<< to resolve):
      - app/main.py
      - docker-compose.yml
```

Resolve conflicts manually:

```python
# app/main.py
<<<<<<< YOUR CHANGES
@app.get("/custom")
def my_endpoint():
    return {"custom": True}
=======
@app.get("/health")
def health_check():
    return {"status": "ok"}
>>>>>>> TEMPLATE CHANGES
```

See `aegis/commands/update.py:460-465` for conflict analysis.
</Accordion>

### Backup and Rollback

Safety mechanisms during updates:

**Automatic Backup** (`aegis/commands/update.py:292-298`):
```bash
aegis update
Creating backup point...
   Backup created: aegis-backup-20260305-143022
```

**Rollback on Failure** (`aegis/commands/update.py:476-487`):
```bash
Update failed: Template version not found

Rollback to previous state? [Y/n]: y
   Rolled back to aegis-backup-20260305-143022
```

**Manual Rollback**:
```bash
# If you decline automatic rollback
git reset --hard aegis-backup-20260305-143022
```

## Component File Management

Understanding what happens to files:

### Adding Components

**Files Added** (`aegis/core/manual_updater.py`):
- Component-specific template files (from `ComponentSpec.template_files`)
- Shared file updates (e.g., `docker-compose.yml`, `pyproject.toml`)

**Files Preserved**:
- All existing files (never overwritten)
- User customizations in shared files

**Configuration Updated**:
```yaml
# .copier-answers.yml
include_scheduler: true  # Added
```

### Removing Components

**Files Deleted**:
- Component-specific files only
- Example: `app/components/scheduler.py`, `app/entrypoints/scheduler.py`

**Files Updated**:
- `docker-compose.yml` - Remove component services
- `pyproject.toml` - Remove component dependencies

**Files Preserved**:
- Database files (e.g., `data/scheduler.db` with scheduler persistence)
- User files not in component template

From `aegis/commands/remove.py:176-186`:
```python
if scheduler_backend == StorageBackends.SQLITE:
    typer.echo("Your scheduler uses SQLite for job persistence.")
    typer.echo("The database file at data/scheduler.db will remain.")
    typer.echo("To remove all data: Also remove the database component")
```

### Updating Templates

**Files Modified**:
- Only files changed in template between versions
- User customizations preserved via 3-way merge

**Files Created**:
- New files added in template version
- New components if enabled in `.copier-answers.yml`

**Files Deleted**:
- Files removed in template version
- Only if not customized by user

## The `.copier-answers.yml` File

The answer file is the key to evolution:

```yaml
# .copier-answers.yml
_commit: abc123def456
_src_path: gh:lbedner/aegis-stack
aegis_version: 0.6.0
project_name: My API
project_slug: my-api

# Component selections
include_scheduler: true
include_worker: true
include_database: true
include_redis: true

# Component configuration
scheduler_backend: sqlite
database_engine: sqlite

# Service selections
include_auth: false
include_ai: false
include_comms: false
```

<Warning>
**Never delete `.copier-answers.yml`** - it's required for `add`, `remove`, and `update` commands to work correctly.
</Warning>

From `copier.yml:17`:
```yaml
_answers_file: .copier-answers.yml
```

## Common Evolution Patterns

### Pattern: Progressive Enhancement

```bash
# Day 1: Ship MVP
aegis init basic-api

# Week 2: Add user accounts
aegis add database

# Month 1: Add background jobs
aegis add worker

# Month 3: Add scheduled tasks
aegis add scheduler

aegis add observability
```

### Pattern: Prototype to Production

```bash
# Prototype: Everything in-memory
aegis init prototype
# No database, no persistence

# Production: Add persistence
aegis add database
aegis add scheduler[sqlite]  # Persistent jobs
# Enable auth service

# Scale: Add workers
aegis add worker
aegis add ingress  # Load balancer
```

### Pattern: Simplification

```bash
# Started with everything
aegis init full-stack --components scheduler,worker,database

# Realized don't need scheduler
aegis remove scheduler

# Realized don't need workers
aegis remove worker
# Auto-removes redis (no longer needed)

# Clean, focused stack
# Components: backend, frontend, database
```

## Why Aegis is Different

From README.md:86-112:

> **Your choices aren't permanent.** Start with what you need today, add components when requirements change, remove what you outgrow.

Most starters lock you in at initialization. Aegis Stack provides:

1. **One-command evolution**: `aegis add`, `aegis remove`, `aegis update`
2. **Git-aware merging**: Safe template updates with conflict detection
3. **Automatic dependency management**: Components auto-add/remove dependencies
4. **Rollback capabilities**: Backup tags enable safe experimentation
5. **No vendor lock-in**: Generated projects work without CLI if needed

## Troubleshooting

<Accordion title="Version mismatch warnings">
**Problem**: CLI version doesn't match template version

**Solution**:
```bash
# Update project to match CLI
aegis update

# Or force operation (not recommended)
aegis add scheduler --force
```
</Accordion>

<Accordion title="Merge conflicts during update">
**Problem**: Update creates `.rej` files with conflicts

**Solution**:
```bash
# Find conflicts
grep -r "<<<<<<<" .

# Resolve manually
vim app/main.py  # Edit conflicted file

# Clean up .rej files
find . -name "*.rej" -delete

# Test and commit
make check
git add .
git commit -m "Resolve merge conflicts from template update"
```
</Accordion>

<Accordion title="Git tree not clean">
**Problem**: `aegis update` requires clean git tree

**Solution**:
```bash
# Commit changes
git add .
git commit -m "WIP: Save before template update"

# Or stash changes
git stash

# Now run update
aegis update

# Restore stash if used
git stash pop
```
</Accordion>

<Accordion title="Component files not removed">
**Problem**: `aegis remove` didn't delete all files

**Solution**:
Component removal only deletes files in `ComponentSpec.template_files`. Other files (like data files) are preserved.

```bash
# Manual cleanup if needed
rm -rf data/scheduler.db

# Review what was removed
git status
```
</Accordion>
